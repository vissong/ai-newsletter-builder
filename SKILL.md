# AI Newsletter Builder

Builds and maintains a static AI newsletter site. It manages pluggable data sources, runs a concurrent collection → merge → categorize → render pipeline, and produces a daily-issue site with an auto-generated index page suitable for GitHub Pages or Cloudflare Pages.

## When to use

Activate when the user wants any of:
- Create a new AI newsletter / digest / briefing site
- Produce "today's issue" or "this week's issue" as a standalone web page
- Add, remove, or configure newsletter data sources (email, webpage, search, CLI)
- Switch or customize the design system for the newsletter
- Publish the site to GitHub Pages or Cloudflare Pages

If the user only wants a one-off text summary with no persistent site, use a plain news briefing instead.

## Project layout this skill builds

All output goes into `<cwd>/site/` by default (ask before writing elsewhere). The layout is stable — every run relies on it:

```
site/
├── index.html                  # homepage, generated by scripts/build_index.py
├── assets/
│   ├── style.css               # derived from the chosen design.md
│   └── ...                     # fonts, icons, images
├── issues/
│   ├── 2026-04-18.html         # one file per daily issue
│   ├── 2026-04-19.html
│   └── ...
├── data/
│   ├── issues.json             # index of all issues (used by build_index.py)
│   └── raw/
│       └── 2026-04-18/         # per-day raw collected content
│           ├── source-a.md
│           └── merged.json     # deduped + categorized items for that day
├── config/
│   ├── sources.yaml            # data source definitions
│   └── design.md               # active design system (copied or symlinked)
└── README.md                   # deployment notes
```

`data/issues.json` is the lightweight manifest the homepage reads. The homepage is always rebuilt from this manifest; the daily HTML files are written once and never touched by the index build. This keeps the index build cheap.

## Workflow overview

```
Initialize (first run only)
  └─ scaffold site/, pick design, seed sources
      ↓
Collect   (concurrent across sources)
  └─ write raw content to site/data/raw/<date>/
      ↓
Merge     (dedup + cross-source fusion)
  └─ one item can have multiple sources; track source_count
      ↓
Categorize (5 fixed buckets)
  └─ 重大发布 / 行业动态及商业价值 / 研究前沿 / 工具发布 / 政策监管
      ↓
Render
  ├─ write site/issues/<date>.html      (full daily page)
  └─ append summary to site/data/issues.json
      ↓
Index
  └─ run scripts/build_index.py to regenerate site/index.html
      ↓
Publish (optional)
  └─ GitHub Pages or Cloudflare Pages
```

The user can also invoke just one step (e.g. "add a new data source" or "regenerate the index"). Don't force the whole pipeline.

## Phase 0: Initialization

Before collecting anything, make sure the site is scaffolded. Check if `site/config/sources.yaml` exists.

If not, walk the user through setup:

1. **Confirm the output dir.** Default `./site`. Ask if unsure.
2. **Pick a design system.** Two formats supported — list both to the user:
   - **Legacy single-file**: `designs/<name>.md` (minimal / editorial / terminal / warm). Token-heavy, Python-parsed, quick.
   - **DESIGN.md (Google Stitch / getdesign.md format)**: `designs/<name>/DESIGN.md`. 9-section prose spec (colors, typography, components, layout, do's & don'ts). Richer, model-interpreted, better for production / brand-match sites.

   Ship-in-the-box DESIGN.md sets live under `designs/<name>/` (currently `figma/`, `cohere/`). Users can also drop their own `DESIGN.md` into a new folder, or fetch one from [getdesign.md](https://getdesign.md) — see `references/design_md_guide.md` §Fetching.
3. **Seed data sources.** Show the built-in source catalog (see `references/data_sources.md`) and ask which to enable. Let the user add their own in the same step (see next phase).
4. **Run** `scripts/init_site.py --dir <cwd>/site --design <name-or-path>` to create the directory skeleton, copy the design into `site/config/design/`, and write an empty `sources.yaml` header. The script detects whether `--design` resolves to a single `.md` or a directory with `DESIGN.md`:
   - Single `.md` → templated CSS gen (fast, built-in).
   - Directory with `DESIGN.md` → script writes a minimal placeholder CSS and **you must then read `DESIGN.md` yourself and write `site/assets/style.css`** following `references/design_md_guide.md`. Don't skip this — the placeholder is ugly by design, to force a proper pass.

After init, confirm with the user before moving to collection — they may want to add more sources first.

## Phase 1: Managing data sources

Sources live in `site/config/sources.yaml`. Four source types are supported:

| Type     | How content is fetched                                                       |
| -------- | ---------------------------------------------------------------------------- |
| `email`  | External email CLI (e.g. `gog` for Gmail). Skill spawns the CLI and parses.  |
| `web`    | Fetched with the available web-reading tool / skill (see below).             |
| `search` | Executes a search query via the available web search tool / skill.           |
| `cli`    | Runs a user-specified CLI command and captures stdout.                       |

For each type, see `references/data_sources.md` for concrete fetch instructions, and `references/add_source_flow.md` for the exact add-source interview and the YAML schema.

### Adding a source (guided flow)

When the user says "add a source" / "接入一个数据源" / similar, run the guided flow from `references/add_source_flow.md`. The short version:

1. Ask for **type** (`email` / `web` / `search` / `cli`).
2. Ask for **name** (used as the directory key — lowercase, dashes).
3. Ask for type-specific parameters (URL, query, command, extraction hints, etc.).
4. **Immediately run a trial fetch** — pull one item and show the extracted fields to the user. Do NOT skip this. A source that looks right in YAML but fails in practice is the single biggest source of silent failures.
5. On success, append to `sources.yaml`. On failure, offer to adjust parameters (selectors, auth, CLI flags) before saving.

### Choosing the web-reading tool

Web and search sources use a **tiered fallback chain**. Try each tier in order; move to the next only when the current one fails or returns empty/blocked content.

**Tier 1 — Real browser (best for JS-heavy or auth-gated pages)**
Check the available skills list. If `web-access`, `browser-use`, or any CDP-capable skill is present, prefer it — it handles JavaScript rendering, cookie persistence, and anti-bot measures.

**Tier 2 — Built-in WebFetch / MCP reader**
Plain HTTP fetch. Works for most server-rendered pages. Tools: built-in `WebFetch`, `mcp__web_reader__webReader`, or similar MCP tools if present.

**Tier 3 — Jina Reader (clean-text fallback)**
When Tiers 1 and 2 both return empty content, a 403/429/paywall, or unreadable HTML, use Jina Reader:
```
GET https://r.jina.ai/<original-url>
```
Jina converts any URL to clean Markdown, bypassing most anti-scraping measures. Works as a plain `WebFetch` call — no special tool needed. Example:
```
https://r.jina.ai/https://venturebeat.com/category/ai/
```

**Tier 4 — Web search fallback**
If all above fail, fall back to `WebSearch` with `site:<domain> AI after:{{yesterday}}` to retrieve recent headlines as search snippets. Lower fidelity but better than nothing.

**Decision log**: Note in `collect.log` which tier was used for each source: `tool: WebFetch (tier-2)` or `tool: jina-reader (tier-3)`. This helps diagnose sources that consistently need a higher tier.

**Consistency within a run**: Prefer using the same tool for all sources of the same type within one collection run (e.g., don't mix Tier 1 for some sources and Tier 3 for others without reason), but do escalate per-source when a lower tier fails.

### CLI sources (including email)

For every `cli` and `email` source, **check the binary is installed before running collection**. A missing CLI should fail loudly with an actionable install hint, not silently skip the source. See `references/data_sources.md` for the exact check pattern per built-in.

## Phase 2: Concurrent collection

For a given `<date>` (default: today, ISO format):

1. Create `site/data/raw/<date>/`.
2. For each enabled source in `sources.yaml`, spawn fetching work in parallel. If the environment has subagents, dispatch one Agent per source. If not, issue tool calls in parallel batches (up to 5 at a time).
3. **Apply a per-source timeout.** A single slow or hung source must not block the whole run. Default timeout: **30 seconds per source** (overridable via `extract.timeout_seconds` in `sources.yaml`). When dispatching a subagent or a `WebFetch`, set an explicit deadline; if it trips, record the source as failed in `collect.log` with `timed_out: true` and move on — don't retry in the same phase.
4. **Default time window:** keep items published within the **last 48 hours**, but mark items from the last **24 hours** as `recent: true`. Render prioritizes `recent: true` items to the top within each category; items older than 48h are dropped unless the user explicitly asks for a wider window (e.g. a weekly digest). Per-source overrides via `extract.time_window_hours` still win.
5. Each source writes a single file `site/data/raw/<date>/<source-name>.md` containing normalized items. Each item block looks like:

   ```markdown
   ## <title>
   - url: https://...
   - source: <source-name>
   - published_at: 2026-04-18T06:00:00Z
   - fetched_at: 2026-04-18T07:10:12Z
   - recent: true   # within last 24h; omit or false for 24–48h

   <2–4 sentence summary of the item, in the user's language>
   ```

6. Write a short `site/data/raw/<date>/collect.log` recording which sources ran, how many items each produced, elapsed time, and any failures (including timeouts).

Concurrency is the whole point of this phase — don't serialize fetches. The timeout is the safety valve that makes concurrency safe.

## Phase 3: Merge and dedupe

Read every `<source>.md` file for the date and build a single `site/data/raw/<date>/merged.json`. An item in `merged.json`:

```json
{
  "id": "stable-slug-or-hash",
  "title": "...",
  "summary": "...",
  "url": "canonical URL (pick the best one)",
  "alt_urls": ["other URLs for the same story"],
  "sources": ["techcrunch-ai", "openai-blog"],
  "source_count": 2,
  "published_at": "2026-04-18T06:00:00Z",
  "category": null
}
```

### Deduplication rules (apply in order)

1. **Same URL** (after stripping tracking params): merge.
2. **Near-identical title** (normalized lowercase, edit distance ≤ 3): merge.
3. **Same core entity + same event type in the same 48h window** (e.g. "OpenAI releases GPT-5.1" across three outlets): merge.

**Dedup runs across all sources at once, not per-category.** Classification happens *after* dedup (Phase 4), so it's perfectly normal — and required — for an item that appeared in a news outlet (heading toward `industry-business`) and a company blog (heading toward `major-release`) to be fused into one item before either category exists. Never dedup within a category; by then it's too late and you'll miss cross-category duplicates (e.g. the same model release written up as both a news story and a research paper).

When merging:
- Keep the most informative summary (usually the longest coherent one; rewrite slightly if summaries conflict on facts — prefer the primary source).
- Combine sources into the `sources` array; `source_count = len(sources)`.
- Use the earliest `published_at`.
- Pick the most authoritative URL (company blog > major outlet > aggregator) as `url`, move the rest to `alt_urls`.

`source_count > 1` is a strong signal of importance — surface it in the rendered page (e.g. "Reported by N sources").

## Phase 4: Categorization

Assign each item to exactly one of these five categories. Names are fixed — don't invent new ones; downstream templates depend on these slugs.

| Slug                | 名称             | 内容                                                                       |
| ------------------- | ---------------- | -------------------------------------------------------------------------- |
| `major-release`     | 重大发布         | 旗舰模型、核心产品首发、里程碑级功能发布                                   |
| `industry-business` | 行业动态及商业价值 | 融资、并购、收入数据、市场格局、B 端落地、政企合作                         |
| `research-frontier` | 研究前沿         | 论文、基准突破、新架构/方法、学术进展                                      |
| `tools-release`     | 工具发布         | 开源项目、SDK、开发者工具、插件、API、agent 框架                           |
| `policy-regulation` | 政策监管         | 法规、政策草案、合规、安全/伦理事件、监管行动                              |

Ambiguous items: pick the category that matches the *primary* news hook, not the secondary angle.

Write the category back into each item in `merged.json`. Then sort within each category by `source_count` desc, then `published_at` desc.

### Cap on `research-frontier` — keep at most 10

Research sources (arxiv, papers-with-code, HuggingFace research posts, MIT Tech Review analysis) flood this category on any given day and push the rest of the issue off the page. After sorting, **trim `research-frontier` to the top 10** by importance. All other categories stay uncapped.

Importance ranking, applied in order — each tie-breaker runs only when the previous one didn't resolve:

1. `source_count` desc — multi-source coverage means the story is breaking out of pure research.
2. Cited in another category's item (cross-referenced): a paper that a product announcement leans on is important.
3. Named lab / institution on a flagship topic: DeepMind, OpenAI, Anthropic, FAIR, Microsoft Research, or top universities on frontier topics (scaling, reasoning, alignment, robotics, multimodal, agentic systems).
4. Benchmark result that beats a well-known prior SOTA by a clear margin (explicit numbers in the summary).
5. `recent: true` (last 24h) beats 24–48h.
6. Alphabetical fallback for full tie.

Record trimmed-out items in `merged.json` with `"trimmed": true` and a one-line `trim_reason` — **do NOT delete them** — so the user can inspect what was dropped and you can restore them if they disagree. Only items with `trimmed != true` render into the issue HTML.

If the user asks for "all research" or a research-focused weekly, raise the cap or disable it for that run.

## Phase 5: Render the daily page

Before rendering, do TWO design-related lookups:

1. **Pick a page layout.** Read `references/newsletter_layouts.md` and pick one of the canonical skeletons (`editorial-longscroll` / `card-grid` / `terminal-log` / `digest-compact`) that matches the active DESIGN.md atmosphere. Record the choice in `site/config/site.yaml` under `layout:` and include an HTML comment `<!-- layout: <name> -->` at the top of every rendered page so future rerenders stay consistent. If `site.yaml` already has a `layout:` set, use that — don't re-pick per run.

2. **Apply the design system.**
   - If `site/config/design/<name>/DESIGN.md` exists → read it first and follow `references/design_md_guide.md`. Make sure `site/assets/style.css` is already the model-written, DESIGN.md-interpreted version (not the init placeholder). If it's still the placeholder, regenerate it now before rendering HTML. The CSS must implement the class names required by the chosen layout — that's the bridge between DESIGN.md (visual language) and newsletter_layouts.md (structure).
   - If `site/config/design.md` (single file) exists → the templated CSS from init is sufficient. Use the plain template output with `card-grid` layout as the default.

Do not strip the §7 "Don't" rules from DESIGN.md while rendering.

For a date `<YYYY-MM-DD>`:

1. Read `site/data/raw/<date>/merged.json`.
2. Read the template `templates/issue.html` and the active design CSS in `site/assets/style.css`.
3. Fill the template (server-side, in-memory) and write to `site/issues/<date>.html`.
4. Append (or update) an entry in `site/data/issues.json`:

   ```json
   {
     "date": "2026-04-18",
     "title": "AI Newsletter · 2026-04-18",
     "path": "issues/2026-04-18.html",
     "item_count": 23,
     "top_items": [
       {"title": "...", "category": "major-release", "source_count": 4},
       {"title": "...", "category": "research-frontier", "source_count": 2}
     ],
     "summary": "今日重点：…（2–3 句）",
     "generated_at": "2026-04-18T08:30:00Z"
   }
   ```

   `top_items` should have 3–5 entries — the homepage uses these as teasers.

Daily pages are self-contained (CSS and any needed JS inlined or loaded from `assets/`). They do NOT read from `issues.json` at view time.

## Phase 6: Rebuild the homepage

The homepage is the only page that aggregates across issues, so it's the only one that needs regeneration after every run. Do it with a dedicated script so the model isn't re-reading every daily page:

```bash
python scripts/build_index.py --site site
```

`build_index.py`:
- Reads `site/data/issues.json`
- Sorts issues descending by date
- Renders `templates/index.html` with the issue manifest
- Writes `site/index.html`

The homepage should show, at minimum: the newsletter title (from `site/config/site.yaml`), the latest issue's summary, a chronological list of all issues with date + item count + category breakdown, and a search/filter widget if the design supports it.

Never hand-edit `site/index.html` — it's a build artifact.

## Phase 7: Publish (optional)

When the user asks to publish, see `references/publish.md` for the two supported targets:

- **GitHub Pages** — commit `site/` (or the repo root) and enable Pages; provide the ready-to-paste workflow.
- **Cloudflare Pages** — connect repo, set build command `python scripts/build_index.py --site site`, output directory `site`.

Always confirm with the user before running `git push` or any command that publishes externally — this is a shared-state action.

## Design system

Two formats supported.

### Legacy single-file — `designs/<name>.md`

Fast, token-only. Parsed by `scripts/init_site.py`. Built-ins:

- `minimal.md` — clean editorial, high whitespace, serif headings
- `editorial.md` — magazine-style with strong typographic contrast
- `terminal.md` — mono, dark, "cyber/technical" feel
- `warm.md` — cream background, soft accent colors, cozy reading feel

Schema in `references/design_systems.md`. Good for quick prototypes and simple one-off sites.

### DESIGN.md (Google Stitch / getdesign.md) — `designs/<name>/DESIGN.md`

Directory-based, 9-section prose spec. This is the richer format — read `references/design_md_guide.md` for the full schema and translation rules. You (the model) interpret DESIGN.md into CSS and component markup; there's no script that parses the prose.

Built-ins under `designs/`:
- `figma/DESIGN.md` — Figma's site: pill buttons, variable-weight type, monochrome chrome with colorful hero gradients
- `cohere/DESIGN.md` — (empty scaffold; drop in a spec or fetch from getdesign.md)

Users can fetch more from [getdesign.md](https://getdesign.md) or write their own.

### How the active design ends up in the site

During init, the chosen design is copied to `site/config/design/`:
- Single-file → `site/config/design.md` (legacy path, kept for back-compat).
- Directory → `site/config/design/<name>/DESIGN.md` plus any other assets from that folder.

Either way, `site/assets/style.css` is the one file every page loads. Regenerate it whenever the design changes.

## Invocation patterns

Match what the user asks to the right subset of phases. You don't always run the full pipeline.

- "创建一个 AI newsletter 站点" → Phase 0, then 1 (seed), then optionally 2–6 for a first issue.
- "添加一个数据源 / add a source" → Phase 1 guided flow only. No rebuild.
- "生成今天的 newsletter" / "出今天的日报" → Phases 2–6 for today's date.
- "补一下昨天的" → Phases 2–6 with `--date` set to yesterday. Leave other issues alone.
- "换个设计风格" → update `site/config/design.md` + regenerate CSS + rerun Phase 6 (and ask whether to re-render existing daily pages).
- "发布到 GitHub Pages" → Phase 7 only, after confirming site already generated.

## Failure modes to watch for

- **Silent source failure.** A misconfigured source can return zero items without raising an error. The `collect.log` must list per-source item counts — check it and warn the user if any source returned 0 for the day.
- **Over-merging.** Aggressive dedup can collapse distinct stories (e.g. two different GPT models announced same day). When in doubt, keep separate; false splits are fixable, lost items aren't.
- **Category drift.** Don't invent new categories mid-session. If an item really doesn't fit, put it under `industry-business` as the general bucket rather than adding a sixth.
- **Homepage bloat.** The homepage loads `issues.json`; keep per-issue entries small (teasers only). Full item lists stay inside the per-day HTML.
