# AI Newsletter Builder

Builds and maintains a static AI newsletter site. It manages pluggable data sources, runs a concurrent collection → merge → categorize → render pipeline, and produces a daily-issue site with an auto-generated index page suitable for GitHub Pages or Cloudflare Pages.

## When to use

Activate when the user wants any of:
- Create a new AI newsletter / digest / briefing site
- Produce "today's issue" or "this week's issue" as a standalone web page
- Add, remove, or configure newsletter data sources (email, webpage, search, CLI)
- Switch or customize the design system for the newsletter
- Publish the site to GitHub Pages or Cloudflare Pages

If the user only wants a one-off text summary with no persistent site, use a plain news briefing instead.

## Project layout this skill builds

All output goes into `<cwd>/site/` by default (ask before writing elsewhere). The layout is stable — every run relies on it:

```
site/
├── index.html                  # SPA homepage shell (JS reads data/issues.json)
├── assets/
│   ├── style.css               # design system CSS
│   └── ...                     # fonts, icons, images
├── issues/
│   └── index.html              # SPA issue shell (JS reads ?date= → merged.json)
├── data/
│   ├── issues.json             # issue manifest (homepage reads this)
│   └── raw/
│       └── 2026-04-18/         # per-day raw collected content
│           ├── follow-builders.json   # one JSON file per source
│           ├── techcrunch-ai.json
│           └── merged.json     # deduped + categorized items (issue page reads this)
├── feed.xml                    # RSS 2.0 feed, generated by scripts/build_feed.py
├── config/
│   ├── sources/                # one folder per data source (auto-discovered)
│   │   ├── follow-builders/
│   │   │   ├── source.yaml     # metadata: type, enabled, priority, ...
│   │   │   └── fetch.py        # script type: fixed fetch script
│   │   ├── techcrunch-ai/
│   │   │   ├── source.yaml
│   │   │   └── prompt.md       # prompt type: LLM fetch+extraction instructions
│   │   └── ...
│   └── design.md               # active design system (copied or symlinked)
└── README.md                   # deployment notes
```

The site uses **client-side rendering**: both `index.html` and `issues/index.html` are static SPA shells with inline JavaScript that fetches JSON data at view time. No server-side HTML generation is needed — the build pipeline only produces JSON files (`issues.json` and `merged.json`). This means changing the design or templates never requires regenerating past issues.

## Workflow overview

```
Initialize (first run only)
  └─ scaffold site/, pick design, copy SPA shells, seed sources
      ↓
Collect   (concurrent across sources)
  └─ write raw content to site/data/raw/<date>/
      ↓
Merge     (dedup + cross-source fusion)
  └─ one item can have multiple sources; track source_count
      ↓
Categorize (5 fixed buckets)
  └─ 重大发布 / 行业动态及商业价值 / 研究前沿 / 工具发布 / 政策监管
      ↓
Manifest
  └─ append/update entry in site/data/issues.json
      ↓
SPA Shells
  └─ ensure site/index.html + site/issues/index.html exist (copy once)
      ↓
RSS Feed
  └─ run scripts/build_feed.py to regenerate site/feed.xml
      ↓
Publish (optional)
  └─ GitHub Pages or Cloudflare Pages
```

The user can also invoke just one step (e.g. "add a new data source" or "regenerate the index"). Don't force the whole pipeline.

## Phase 0: Initialization

Before collecting anything, make sure the site is scaffolded. Check if `site/config/sources/` exists and contains at least one source folder (a subdirectory with a `source.yaml`).

If not, walk the user through setup:

1. **Confirm the output dir.** Default `./site`. Ask if unsure.
2. **Pick a design system.** Two formats supported — list both to the user:
   - **Legacy single-file**: `designs/<name>.md` (minimal / editorial / terminal / warm). Token-heavy, Python-parsed, quick.
   - **DESIGN.md (Google Stitch / getdesign.md format)**: `designs/<name>/DESIGN.md`. 9-section prose spec (colors, typography, components, layout, do's & don'ts). Richer, model-interpreted, better for production / brand-match sites.

   Ship-in-the-box DESIGN.md sets live under `designs/<name>/` (currently `figma/`, `cohere/`). Users can also drop their own `DESIGN.md` into a new folder, or fetch one from [getdesign.md](https://getdesign.md) — see `references/design_md_guide.md` §Fetching.
3. **Seed data sources.** Show the built-in source catalog (see `references/data_sources.md`) and ask which to enable. Let the user add their own in the same step (see next phase).
4. **Check optional dependencies.** Based on the sources the user chose, check whether external tools are needed and guide installation:

   **Tavily CLI (`tvly`)** — required by all `search-*` sources. Check:
   ```bash
   which tvly || echo "not installed"
   ```
   If missing, guide the user:
   > Search 类数据源需要 Tavily CLI。安装：
   > ```
   > curl -fsSL https://cli.tavily.com/install.sh | bash && tvly login
   > ```
   > 免费版每月 1000 次搜索，足够日常使用。安装完成后告诉我继续。

   **browser-use (Playwright)** — required by sources with `requires_capability: real_browser` (e.g. `venturebeat-ai`, `theverge-ai`, `36kr-ai`). Check:
   ```bash
   test -x ~/.browser-use-env/bin/python3 && echo "ok" || echo "not installed"
   ```
   If missing, guide the user:
   > 部分数据源需要 headless 浏览器抓取（反爬/JS 渲染站点）。安装：
   > ```
   > uv venv ~/.browser-use-env --python 3.13
   > uv pip install --python ~/.browser-use-env/bin/python3 browser-use playwright
   > ~/.browser-use-env/bin/python3 -m playwright install chromium

   > 安装完成后告诉我继续。

   ```bash
   which gog || echo "not installed"
   ```
   If missing, guide the user to install from [gogcli.sh](https://gogcli.sh) and run `gog auth login`.

   Skip any dependency check if the user didn't select sources that need it. Don't block init for optional dependencies — just warn and note which sources will be skipped at collection time.

5. **Run** `scripts/init_site.py --dir <cwd>/site --design <name-or-path> [--seed-sources follow-builders,...]` to create the directory skeleton, copy the design into `site/config/design/`, and seed source folders under `site/config/sources/`. The script detects whether `--design` resolves to a single `.md` or a directory with `DESIGN.md`:
   - Single `.md` → templated CSS gen (fast, built-in).
   - Directory with `DESIGN.md` → script writes a minimal placeholder CSS and **you must then read `DESIGN.md` yourself and write `site/assets/style.css`** following `references/design_md_guide.md`. Don't skip this — the placeholder is ugly by design, to force a proper pass.

After init, confirm with the user before moving to collection — they may want to add more sources first.

## Phase 1: Managing data sources

Each data source is a **folder** under `site/config/sources/<name>/`. The pipeline auto-discovers all folders containing a `source.yaml`. Two source types are supported:

| Type     | How content is fetched                                                           |
| -------- | -------------------------------------------------------------------------------- |
| `script` | A fixed script (`fetch.py`/`fetch.sh`) with stable data format; outputs JSON to stdout. |
| `prompt` | An LLM prompt (`prompt.md`) describing how to fetch and extract; for sources without stable formats. |

For the full schema and examples, see `references/source_extension_guide.md`. For the built-in source catalog, see `references/data_sources.md`. For the add-source interview, see `references/add_source_flow.md`.

### Adding a source (guided flow)

When the user says "add a source" / "接入一个数据源" / similar, run the guided flow from `references/add_source_flow.md`. The short version:

1. Ask for **type** (`script` or `prompt`). Rule of thumb: stable API/CLI → script; web pages that change layout → prompt.
2. Ask for **name** (used as the folder name — lowercase, dashes).
3. Ask for type-specific parameters and co-write the fetch script or prompt.md.
4. **Immediately run a trial fetch** — pull a few items and show the extracted fields to the user. Do NOT skip this.
5. On success, create the source folder under `site/config/sources/<name>/`. On failure, adjust and retry before saving.

### Fetch strategy for prompt-based sources

Prompt-based sources with `fetch_method: web` should follow the **RSS-first strategy** and **tiered fallback chain** described in the source's `prompt.md`. The general guidance:

**RSS-first**: prefer RSS feeds when available — they're more reliable and cheaper. Fall back to web scraping only when there's no RSS, the RSS is stale, or it returns empty content for 2+ runs.

**Tiered fallback chain** for web fetching:

| Tier | Tool | When to use |
|---|---|---|
| 1 | `web-access` / `browser-use` skill (real browser) | JS-rendered, login walls, anti-bot |
| 2 | Built-in `WebFetch` / MCP reader | Standard server-rendered pages |
| 3 | Jina Reader (`https://r.jina.ai/<url>`) | Tiers 1-2 fail or return blocked content |
| 4 | `WebSearch` with `site:<domain>` | All above fail; headlines-only fallback |

Log the tier used per source in `collect.log`.

### Script-based sources

For every `script` source, the collector checks `check.binary` (if specified) before running the script. A missing binary should fail loudly with the `install_hint`, not silently skip the source.

## Phase 2: Concurrent collection

For a given `<date>` (default: today, ISO format):

1. Create `site/data/raw/<date>/`.
2. **Auto-discover sources**: scan `site/config/sources/` for subdirectories containing `source.yaml`. Read each `source.yaml`; skip if `enabled: false`.
3. For each enabled source, dispatch based on type — in parallel. If the environment has subagents, dispatch one Agent per source. If not, issue tool calls in parallel batches (up to 5 at a time).

   **`type: script`** — run the fetch script:
   - Check `check.binary` if specified; fail with `install_hint` if missing.
   - Resolve template variables in `args` (`{{date}}`, `{{recent_hours}}`, etc.).
   - Execute: `<runtime> site/config/sources/<name>/<script> [--key value ...]`
   - Capture stdout (must be a JSON array), validate it parses as JSON.
   - Write to `site/data/raw/<date>/<name>.json`.

   **`type: prompt`** — LLM-driven fetch:
   - Read `site/config/sources/<name>/prompt.md` for fetch + extraction instructions.
   - Read `source.yaml` for `extract` config (item_limit, time_window, exclude_keywords, etc.).
   - Execute the instructions using available tools, following the tiered fallback chain.
   - Format results as a JSON array matching the output contract.
   - Write to `site/data/raw/<date>/<name>.json`.

4. **Apply a per-source timeout.** Default: **30 seconds** (overridable via `timeout_seconds` in `source.yaml`). Timed-out sources are marked failed in `collect.log` — don't retry in the same phase.
5. **Post-processing** by the collector for each source's output:
   - Inject `"source": "<name>"` into each item.
   - Inject `"fetched_at": "<now>"` into each item.
   - Apply `time_window_hours`: drop items older than the window.
   - Apply `recent_hours`: mark items within the window as `"recent": true`.
6. Write a short `site/data/raw/<date>/collect.log` recording which sources ran, how many items each produced, elapsed time, and any failures (including timeouts).

Concurrency is the whole point of this phase — don't serialize fetches. The timeout is the safety valve that makes concurrency safe.

## Phase 3: Merge and dedupe

Read every `<source>.json` file for the date and build a single `site/data/raw/<date>/merged.json`. For backward compatibility, also check for `.md` files (old format) and parse them with the legacy Markdown parser if no `.json` exists for that source. An item in `merged.json`:

```json
{
  "id": "stable-slug-or-hash",
  "title": "...",
  "summary": "...",
  "url": "canonical URL (pick the best one)",
  "alt_urls": ["other URLs for the same story"],
  "sources": ["techcrunch-ai", "openai-blog"],
  "source_count": 2,
  "published_at": "2026-04-18T06:00:00Z",
  "category": null
}
```

### Deduplication rules (apply in order)

1. **Same URL** (after stripping tracking params): merge.
2. **Near-identical title** (normalized lowercase, edit distance ≤ 3): merge.
3. **Same core entity + same event type in the same 48h window** (e.g. "OpenAI releases GPT-5.1" across three outlets): merge.

**Dedup runs across all sources at once, not per-category.** Classification happens *after* dedup (Phase 4), so it's perfectly normal — and required — for an item that appeared in a news outlet (heading toward `industry-business`) and a company blog (heading toward `major-release`) to be fused into one item before either category exists. Never dedup within a category; by then it's too late and you'll miss cross-category duplicates (e.g. the same model release written up as both a news story and a research paper).

When merging:
- Keep the most informative summary (usually the longest coherent one; rewrite slightly if summaries conflict on facts — prefer the primary source).
- Combine sources into the `sources` array; `source_count = len(sources)`.
- Use the earliest `published_at`.
- Pick the most authoritative URL (company blog > major outlet > aggregator) as `url`, move the rest to `alt_urls`.

`source_count > 1` is a strong signal of importance — surface it in the rendered page (e.g. "Reported by N sources").

## Phase 3.5: Translate to output language

After merging, translate all items to the site's `output_language` (from `site/config/site.yaml`). This ensures every item renders consistently in the target language regardless of its original source language.

**When `output_language: zh`:**
- Items with `language: en` (or any non-zh language): translate `title` and `summary` to Chinese. Keep the original `url` intact. Store the original English title in `title_original` for reference.
- Items already in Chinese: no change.
- Translation should be natural and idiomatic, not word-for-word. Proper nouns (model names, company names, product names) keep their original form — e.g. "Claude Opus 4.7" stays as-is, "OpenAI" stays as-is.

**When `output_language: en`:**
- Mirror the above: translate zh items to English.

This step runs on the full `merged.json` before categorization (Phase 4). It can be done in a single batch pass.

## Phase 4: Categorization

Assign each item to exactly one of these five categories. Names are fixed — don't invent new ones; downstream templates depend on these slugs.

| Slug                | 名称             | 内容                                                                       |
| ------------------- | ---------------- | -------------------------------------------------------------------------- |
| `major-release`     | 重大发布         | 旗舰模型、核心产品首发、里程碑级功能发布                                   |
| `industry-business` | 行业动态及商业价值 | 融资、并购、收入数据、市场格局、B 端落地、政企合作                         |
| `research-frontier` | 研究前沿         | 论文、基准突破、新架构/方法、学术进展                                      |
| `tools-release`     | 工具发布         | 开源项目、SDK、开发者工具、插件、API、agent 框架                           |
| `policy-regulation` | 政策监管         | 法规、政策草案、合规、安全/伦理事件、监管行动                              |

Ambiguous items: pick the category that matches the *primary* news hook, not the secondary angle.

Write the category back into each item in `merged.json`. Then sort within each category by `source_count` desc, then `published_at` desc.

### Per-category caps

To keep daily issues readable, apply per-category caps after sorting. Excess items are trimmed, not deleted.

| Category            | Max visible | Reason                                              |
| ------------------- | ----------- | --------------------------------------------------- |
| `research-frontier` | 10          | arXiv alone can flood 90+ items/day                 |
| `major-release`     | 20          | Prevent noise from minor version bumps              |
| `industry-business` | 20          | High volume from search + tencent-news sources      |
| `tools-release`     | 20          | follow-builders + open-source sources produce many  |
| `policy-regulation` | 20          | Keeps balance with other categories                 |

Research sources (arxiv, papers-with-code, HuggingFace research posts, MIT Tech Review analysis) flood this category on any given day and push the rest of the issue off the page. After sorting, **trim `research-frontier` to the top 10** by importance. Other categories trim to **20** each.

Importance ranking, applied in order — each tie-breaker runs only when the previous one didn't resolve:

1. `source_count` desc — multi-source coverage means the story is breaking out of pure research.
2. Cited in another category's item (cross-referenced): a paper that a product announcement leans on is important.
3. Named lab / institution on a flagship topic: DeepMind, OpenAI, Anthropic, FAIR, Microsoft Research, or top universities on frontier topics (scaling, reasoning, alignment, robotics, multimodal, agentic systems).
4. Benchmark result that beats a well-known prior SOTA by a clear margin (explicit numbers in the summary).
5. `recent: true` (last 24h) beats 24–48h.
6. Alphabetical fallback for full tie.

Record trimmed-out items in `merged.json` with `"trimmed": true` and a one-line `trim_reason` — **do NOT delete them** — so the user can inspect what was dropped and you can restore them if they disagree. Only items with `trimmed != true` render into the issue HTML.

If the user asks for "all research" or a research-focused weekly, raise the cap or disable it for that run.

## Phase 5: Update the issue manifest

After categorization, update `site/data/issues.json` with metadata for the new issue. **No HTML is generated** — the site uses client-side rendering (SPA shells that read JSON at view time).

For a date `<YYYY-MM-DD>`:

1. Read `site/data/raw/<date>/merged.json` to compute statistics.
2. Append (or update) an entry in `site/data/issues.json`:

   ```json
   {
     "date": "2026-04-18",
     "title": "Mythos 走进白宫，Claude Design 挑战 Figma",
     "item_count": 23,
     "source_count": 8,
     "categories": {
       "major-release": 3,
       "industry-business": 7,
       "research-frontier": 5,
       "tools-release": 4,
       "policy-regulation": 4
     },
     "top_items": [
       {"title": "...", "id": "...", "category": "major-release", "source_count": 4},
       {"title": "...", "id": "...", "category": "research-frontier", "source_count": 2}
     ],
     "summary": "今日重点：…（2–3 句）",
     "n": 1,
     "dow": "FRI",
     "generated_at": "2026-04-18T08:30:00Z"
   }
   ```

   ### Issue title — 每期标题

   `title` 必须是**当期独有的编辑标题**，不能用模板化的 `站点名 · 日期` 格式。好的标题像新闻头版标题一样，概括当天最重要的 1–2 个事件，让读者一眼知道这期的看点。

   生成规则：
   1. 从 `top_items`（`source_count` 最高的条目）和 `major-release` 分类中提取当天最有新闻价值的 1–2 条。
   2. 用一句话概括，不超过 30 个中文字符（或 60 个英文字符）。
   3. 可以用顿号或逗号连接两个关键词，但不要堆砌。
   4. 不要包含日期——日期已有 `date` 字段。
   5. 不要包含站点名——那是站点层面的信息，不是期刊层面的。

   示例：
   - ✅ `"Mythos 走进白宫，Claude Design 挑战 Figma"`
   - ✅ `"GPT-5 发布，开源大模型性能首次追平闭源"`
   - ✅ `"欧盟 AI 法案正式生效"`
   - ❌ `"AI Newsletter · 2026-04-18"`（模板化）
   - ❌ `"2026年4月18日 AI 日报"`（含日期）

   Fields used by the SPA homepage:
   - `date` — sort key and URL parameter (`issues/?date=2026-04-18`)
   - `categories` — object mapping slug → count (used for filter chips and stats)
   - `source_count` — total unique data sources
   - `top_items` — 3–5 entries for homepage teasers (include `id` for anchor links)
   - `headlines` — exactly 2 editorial headlines (see below)
   - `n` — issue number (1-based, ascending)
   - `dow` — day-of-week abbreviation (`SUN`..`SAT`)

   ### Headlines — 首页头条

   每期在 `issues.json` 中生成 `headlines` 数组，包含 **2 条**编辑式头条，展示在首页"最新一期"区域的右侧。头条不是简单复制 `top_items` 的标题——它们是对当天最重要事件的**提炼和再表达**。

   ```json
   "headlines": [
     {"title": "Anthropic 三线齐发：Claude Design 进军设计、Opus 4.7 夺回性能王座", "tag": "重大发布 · 2 源报道"},
     {"title": "Mythos 秘密入驻白宫与 NSA，AI 国家安全博弈升级", "tag": "政策监管"}
   ]
   ```

   生成规则：
   1. 从当天所有可见条目中选出**最有新闻价值的 2 个主题**（不限于同一分类）。优先选 `source_count` 高的、跨分类影响大的事件。
   2. 每条头条是一句**完整的、有信息量的中文短句**（15–30 字），而非单纯的标题复制。要让读者不看正文也能抓住核心。
   3. `tag` 字段标注分类名称，如果 `source_count > 1` 则附加 `· N 源报道`。
   4. 两条头条应覆盖**不同的分类或主题方向**，避免同质化。

## Phase 6: Ensure SPA shells are in place

The site uses two static HTML shell templates that render content client-side via JavaScript:

- **`site/index.html`** — fetches `data/issues.json`, renders the homepage with issue list, search, and filters
- **`site/issues/index.html`** — reads `?date=` URL parameter, fetches `../data/raw/<date>/merged.json`, renders categorized articles

Both templates are copied from `templates/index.html` and `templates/issue.html` during `init_site.py`. The `{{ title }}` and `{{ subtitle }}` placeholders are replaced once at copy time.

Run the build script to ensure templates are in place (idempotent, only copies if missing):

```bash
python scripts/build_index.py --site site
```

`build_index.py`:
- Reads `site/config/site.yaml` for title/subtitle
- Copies `templates/index.html` → `site/index.html` (if missing)
- Copies `templates/issue.html` → `site/issues/index.html` (if missing)
- Replaces `{{ title }}` and `{{ subtitle }}` during copy

**Routing**: All designs use the same SPA approach. No per-design exceptions.
- Homepage: `site/index.html` → JS fetches `data/issues.json`
- Issue page: `site/issues/index.html?date=<YYYY-MM-DD>` → JS fetches `../data/raw/<date>/merged.json`

Never hand-edit `site/index.html` or `site/issues/index.html` — they are copied from templates.

**CSS**: The design system CSS must exist at `site/assets/style.css` before the pages will render correctly. For DESIGN.md designs, ensure the CSS has been generated from the design spec. For designs with a pre-built `style.css` (like `ciyuan-jie`), copy it during init.

## Phase 6.5: Regenerate RSS feed

Run immediately after Phase 6 (index rebuild). The feed is a build artifact like `index.html` — never hand-edit it.

```bash
python scripts/build_feed.py --site site --base-url https://your-site.example.com
```

`build_feed.py`:
- Reads `site/data/issues.json` (same manifest as the homepage)
- Sorts issues descending by date
- Writes `site/feed.xml` (RSS 2.0 with Atom self-link)

**`--base-url` is required for feed readers** — relative URLs in `<link>` and `<guid>` break every reader. Read `base_url` from `site/config/site.yaml` if the user has set it there (`base_url: https://...`); otherwise prompt once and save it back to `site.yaml`.

Each `<item>` in the feed contains:
- `<title>` — issue title
- `<link>` / `<guid>` — absolute URL to the issue page (`<base_url>/issues/?date=<YYYY-MM-DD>`)
- `<pubDate>` — RFC-822 timestamp from `generated_at`
- `<description>` — CDATA block with the full categorized content. Data source:
  - `site/data/raw/<date>/merged.json` — full structured items grouped by category
  - `issues.json` `top_items` — fallback summary-only if merged.json is missing

  The rich format shows each category as an `<h3>` heading followed by a `<ul>` with every item's title (linked), multi-source badge, and summary sentence. This gives feed readers the complete daily content without visiting the site.

The SPA shell templates already include the feed autodiscovery `<link>` tag — no post-processing needed.

**When to skip:** If `issues.json` has no entries yet, skip silently (the script exits with a clear error; don't surface it to the user as a failure).

## Phase 7: Publish (optional)

When the user asks to publish, see `references/publish.md` for the two supported targets:

- **GitHub Pages** — commit `site/` (or the repo root) and enable Pages; provide the ready-to-paste workflow.
- **Cloudflare Pages** — connect repo, set build command `python scripts/build_index.py --site site`, output directory `site`.

Always confirm with the user before running `git push` or any command that publishes externally — this is a shared-state action.

## Design system

Two formats supported.

### Legacy single-file — `designs/<name>.md`

Fast, token-only. Parsed by `scripts/init_site.py`. Built-ins:

- `minimal.md` — clean editorial, high whitespace, serif headings
- `editorial.md` — magazine-style with strong typographic contrast
- `terminal.md` — mono, dark, "cyber/technical" feel
- `warm.md` — cream background, soft accent colors, cozy reading feel

Schema in `references/design_systems.md`. Good for quick prototypes and simple one-off sites.

### DESIGN.md (Google Stitch / getdesign.md) — `designs/<name>/DESIGN.md`

Directory-based, 9-section prose spec. This is the richer format — read `references/design_md_guide.md` for the full schema and translation rules. You (the model) interpret DESIGN.md into CSS and component markup; there's no script that parses the prose.

Built-ins under `designs/`:
- `figma/DESIGN.md` — Figma's site: pill buttons, variable-weight type, monochrome chrome with colorful hero gradients
- `cohere/DESIGN.md` — (empty scaffold; drop in a spec or fetch from getdesign.md)
- `ciyuan-jie/DESIGN.md` — 词元长街 · Token Street: Chinese newspaper broadsheet aesthetic, warm paper + 朱砂红 accent + Noto Serif SC / Playfair Display / JetBrains Mono. **CSS is pre-built** — `designs/ciyuan-jie/style.css` should be copied verbatim to `site/assets/style.css`; do not regenerate from DESIGN.md. Pair with `editorial-longscroll` layout.

Users can fetch more from [getdesign.md](https://getdesign.md) or write their own.

### How the active design ends up in the site

During init, the chosen design is copied to `site/config/design/`:
- Single-file → `site/config/design.md` (legacy path, kept for back-compat).
- Directory → `site/config/design/<name>/DESIGN.md` plus any other assets from that folder.

Either way, `site/assets/style.css` is the one file every page loads. Regenerate it whenever the design changes.

## Invocation patterns

Match what the user asks to the right subset of phases. You don't always run the full pipeline.

- "创建一个 AI newsletter 站点" → Phase 0, then 1 (seed), then optionally 2–6 for a first issue.
- "添加一个数据源 / add a source" → Phase 1 guided flow only. No rebuild.
- "生成今天的 newsletter" / "出今天的日报" → Phases 2–6.5 for today's date.
- "补一下昨天的" → Phases 2–6.5 with `--date` set to yesterday. Leave other issues alone.
- "换个设计风格" → update `site/config/design.md` + regenerate CSS + rerun Phase 6 (and ask whether to re-render existing daily pages).
- "生成 RSS feed" / "重新生成 feed.xml" → Phase 6.5 only (run `build_feed.py`).
- "发布到 GitHub Pages" → Phase 7 only, after confirming site already generated.

## Failure modes to watch for

- **Silent source failure.** A misconfigured source can return zero items without raising an error. The `collect.log` must list per-source item counts — check it and warn the user if any source returned 0 for the day.
- **Over-merging.** Aggressive dedup can collapse distinct stories (e.g. two different GPT models announced same day). When in doubt, keep separate; false splits are fixable, lost items aren't.
- **Category drift.** Don't invent new categories mid-session. If an item really doesn't fit, put it under `industry-business` as the general bucket rather than adding a sixth.
- **Homepage bloat.** The homepage loads `issues.json`; keep per-issue entries small (teasers only). Full item lists stay inside the per-day HTML.
